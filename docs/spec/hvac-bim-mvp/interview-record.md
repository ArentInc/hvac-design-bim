# hvac-bim-mvp ヒアリング記録

**作成日**: 2026-03-26
**ヒアリング実施**: step4 既存情報ベースの差分ヒアリング

## ヒアリング目的

PRD v2.4 の内容と既存コードベースの分析結果を基に、未定義・曖昧な部分の明確化と、PRDからの方針変更を確認するためのヒアリングを実施しました。

## 質問と回答

### Q1: PRD v2.4 の承認状況

**質問日時**: 2026-03-26
**カテゴリ**: 既存設計確認
**背景**: PRDが複数回改訂（v1.0→v2.4）されており、最終版として扱えるか確認が必要

**回答**: v2.4で確定

**信頼性への影響**:
- この回答により、PRD記載の全要件が信頼性レベル 🔵 の基盤として使用可能に
- PRD内容に基づく要件は全て「確実な要件」として定義可能

---

### Q2: ZoneNode と HvacZoneNode の関係

**質問日時**: 2026-03-26
**カテゴリ**: 未定義部分詳細化
**背景**: 既存の ZoneNode（建築用ポリゴンゾーン）と、PRD定義の HvacZoneNode（空調用ゾーン）が機能的に類似しており、統合/分離の方針が不明

**回答**: 完全分離

**信頼性への影響**:
- REQ-201（HvacZoneNode設計）の信頼性が 🟡 → 🔵 に向上
- 既存ZoneNodeへの影響がないことが確定し、AnyNode拡張のリスクが低下

---

### Q3: 開発フェーズの実施順序

**質問日時**: 2026-03-26
**カテゴリ**: 優先順位
**背景**: PRD定義の5フェーズ（MVP-0〜MVP-4）が実務的に最適かの確認

**回答**: PRDのフェーズ通り（MVP-0 → MVP-1 → MVP-2 → MVP-3 → MVP-4）

**信頼性への影響**:
- 開発計画関連の全要件が 🔵 として確定
- フェーズ依存関係がPRD通りであることが保証

---

### Q4: Supabase 統合タイミング

**質問日時**: 2026-03-26
**カテゴリ**: 追加要件
**背景**: PRD v2.3/v2.4 では Supabase を MVP-0 で導入と定義しているが、実際の優先度を確認

**回答**: MVP後に導入

**信頼性への影響**:
- PRDセクション18.3〜18.5（Supabase関連）は MVP スコープ外に変更
- 永続化要件が IndexedDB のみに簡素化（REQ-1301 の信頼性: 🔵）
- MCP によるインフラセットアップ（セクション18.5）は要件定義から除外

---

### Q5: エディタフェーズの共存方法

**質問日時**: 2026-03-26
**カテゴリ**: 未定義部分詳細化
**背景**: 既存の3フェーズ（site/structure/furnish）と HVAC の4フェーズ（zone/equip/route/calc）をどう組み合わせるか未定義

**回答**: モード切替（「建築モード」と「HVACモード」の2モード制）

**信頼性への影響**:
- 新規要件 REQ-1401（モード切替UI）を追加（信頼性: 🔵）
- useEditor ストアの Phase 型定義に影響（建築用フェーズとHVAC用フェーズを分離）
- 既存の site/structure/furnish フェーズは変更なし

---

### Q6: 建築参照モデルの読込方法

**質問日時**: 2026-03-26
**カテゴリ**: 未定義部分詳細化
**背景**: PRDでは ArchitectureRefNode として参照メッシュ/簡易IFC読込と定義しているが、既存の建築データとの関係が不明

**回答**: 両方対応（既存シーン参照 + サンプルJSON読込）

**信頼性への影響**:
- REQ-101（建築参照読込）が2つのサブ要件に分割（信頼性: 🔵）
- 既存Pascal Editorで作成した建築データの活用が可能に

---

### Q7: Must Have 要件の確認

**質問日時**: 2026-03-26
**カテゴリ**: 優先順位
**背景**: PRD定義の機能領域のうち、MVPで必須の範囲を確認

**回答**: 全4領域が Must Have
- ゾーニング + 負荷計算
- 系統構成 + 機器選定
- ダクトルーティング + 計算
- 配管ルーティング

**信頼性への影響**:
- PRDセクション9.3（必須要求）の全項目が 🔵 として確定
- 配管ルーティングも必須であることが再確認

---

### Q8: デモシナリオのスコープ

**質問日時**: 2026-03-26
**カテゴリ**: 追加要件
**背景**: PRDセクション21のデモ仕様が要件定義のスコープに入るか確認

**回答**: 含める

**信頼性への影響**:
- デモ関連要件（プリセットデータ、画面表示仕様、操作フィードバック）が全て 🔵 として確定
- MVP-4 フェーズの作業量が明確化

---

### Q9: 保存方式

**質問日時**: 2026-03-26
**カテゴリ**: 影響範囲
**背景**: Supabase MVP後導入の決定を受け、保存方式を確認

**回答**: IndexedDBのみ

**信頼性への影響**:
- 保存/再読込の要件が簡素化（IndexedDB永続化のみ）
- JSONエクスポート/インポートはMVPスコープ外

### Q10: スキーマバージョニング/migration runner

**質問日時**: 2026-03-26
**カテゴリ**: 影響範囲
**背景**: PRDではMVP-0で必須とされるが、Supabase後回しの影響を確認

**回答**: 後回し

**信頼性への影響**:

---

### Q11: 既存機能への影響許容度

**質問日時**: 2026-03-26
**カテゴリ**: 影響範囲
**背景**: HVAC機能追加に伴うAnyNode拡張、Phase型変更等が既存機能に与える影響を確認

**回答**: 許容する（必要なリファクタリングもOK）

**信頼性への影響**:
- 実装上の制約が緩和され、最適なアーキテクチャ設計が可能に
- 既存コード変更を伴う要件も 🔵 として定義可能

## ヒアリング結果サマリー

### 確認できた事項
- PRD v2.4 が確定版として使用可能
- ZoneNode と HvacZoneNode は完全分離
- 開発フェーズは PRD 通り（MVP-0〜MVP-4）
- 全4機能領域が Must Have
- デモシナリオもスコープ内
- 既存コードへのリファクタリングは許容

### PRDからの変更要件
- Supabase 統合を MVP 後に延期（IndexedDB のみ）
- スキーマバージョニング/migration runner を後回し
- 「建築モード」「HVACモード」の2モード制を新規追加
- 建築参照は既存シーン参照 + サンプルJSON の両方対応

### 残課題
- DuctFittingNode（エルボ、T分岐等）の詳細仕様がPRDで薄い
- HVACモードと建築モードの切替UI詳細が未定義
- 既存 ZoneNode の HVAC ゾーンとしての参照連携の可否
