## 実装詳細

### 1. 🔵 buildSystemGraph -- 系統グラフ構築

**ファイル**: `packages/core/src/utils/hvac-graph.ts`

architecture.md のグラフ構築アルゴリズムに基づく。

```typescript
interface SystemGraph {
  nodes: Map<string, GraphNode>       // nodeId → GraphNode
  edges: Map<string, GraphEdge>       // edgeId → GraphEdge
  root: string | null                 // AHU nodeId（根ノード）
  leaves: string[]                    // Diffuser nodeIds（葉ノード）
}

interface GraphNode {
  nodeId: string
  nodeType: string                    // 'ahu' | 'diffuser' | 'duct_segment' | 'duct_fitting' | 'pipe_segment'
  ports: Map<string, string | null>   // portId → 接続先nodeId（null=未接続）
  neighbors: string[]                 // 隣接nodeId配列
}

interface GraphEdge {
  id: string
  from: string                        // 始点nodeId
  to: string                          // 終点nodeId
  fromPortId: string
  toPortId: string
}
```

**関数シグネチャ**:
```typescript
function buildSystemGraph(
  systemId: string,
  nodes: Record<string, AnyNode>
): SystemGraph
```

処理:
1. systemId に属する全ノードを収集
2. 各ノードのポートから接続関係を構築
3. AHUを根、Diffuserを葉として識別
4. 隣接リストを生成

### 2. 🟡 BFS/DFS トラバーサル関数

ポートIDを辞書でたどり、隣接ノードを順に訪問する。

```typescript
function traverseBFS(
  graph: SystemGraph,
  startNodeId: string,
  visitor: (nodeId: string, depth: number) => boolean | void
): void

function traverseDFS(
  graph: SystemGraph,
  startNodeId: string,
  visitor: (nodeId: string, depth: number) => boolean | void
): void
```

- visitor が `false` を返した場合、そのブランチの探索を中止
- 訪問済みセットで重複訪問を防止

### 3. 🟡 サイクル検出

EDGE-004（循環参照検出）への対応。

```typescript
function detectCycles(graph: SystemGraph): CycleResult

interface CycleResult {
  hasCycle: boolean
  cycleNodes: string[][]              // サイクルを構成するnodeId配列（複数サイクル可能）
}
```

処理:
- DFSベースのサイクル検出（visiting/visited 2状態管理）
- サイクル発見時、関与ノードIDを記録

### 4. 🟡 最遠経路探索

圧損計算用の最長パス（AHU→最遠Diffuser）を特定。

```typescript
function findLongestPath(
  graph: SystemGraph
): PathResult

interface PathResult {
  path: string[]                      // AHU → ... → Diffuser のnodeId配列
  segmentCount: number                // 経路上のセグメント数
  totalLength: number                 // 経路上のダクト/配管総延長 (m)
}
```

処理:
- 根(AHU)から全葉(Diffuser)へのパスを列挙
- セグメント総延長が最大のパスを選択

### 5. 🔵 逆トラバーサル -- 葉→根の風量合算

REQ-801~802 に基づく。Diffuser(葉)からAHU(根)へ向かって風量を合算する。

```typescript
function aggregateAirflow(
  graph: SystemGraph,
  nodes: Record<string, AnyNode>
): Map<string, number>               // segmentId → aggregatedAirflow
```

処理:
1. 全Diffuserの設計風量を取得
2. 葉→根方向にトラバース
3. 分岐点でブランチ風量を合算
4. 各セグメントに通過風量を記録

## 注意事項

- **純粋関数**: hvac-graph.ts はステートレスなユーティリティ関数群。Zustandストアへの依存なし。引数としてノードマップを受け取る
- **ポートID辞書**: 接続はポートIDで解決する。segmentのstartPortId/endPortIdとfittingのportsが対応。connectedSegmentIdで逆引き
- **パフォーマンス**: MVP段階では100ノード以下を想定。O(V+E)のBFS/DFSで十分。将来的に大規模対応が必要な場合は最適化
- **テスト用モックデータ**: テストではfactory関数を用意し、ノード間の接続関係を簡潔に記述できるようにする
- **配管グラフ**: PipeSegmentもダクトと同じグラフ構造。medium（chilled_water等）でフィルタリングして別グラフとして構築可能
