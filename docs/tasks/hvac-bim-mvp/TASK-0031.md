## タスク概要

AHU起点の接続グラフをトラバースし、各DuctSegmentのairflowRateを下流合算で自動設定するシステムを実装する。hvac-graph.tsのbuildSystemGraphを使用してグラフ構造を取得し、葉ノード（Diffuser）から根ノード（AHU）へ逆トラバースで各区間の風量を合算する。

## 実装詳細

### 1. 🔵 AirflowDistributionSystemコンポーネント作成

**ファイル**: `packages/core/src/systems/hvac/airflow-distribution-system.tsx`

Coreシステムとして実装。Reactコンポーネントとして`null`を返し、useFrame内でdirtyNodesを監視して風量配分を再計算する。

- dirtyNodesにDuctSegment/Diffuser/AHUが含まれる場合に再計算トリガー
- 該当systemIdの全グラフを再計算

### 2. 🔵 hvac-graph.tsのbuildSystemGraph使用

hvac-graph.tsが提供するグラフ構築関数を利用してダクト接続グラフを取得する。

- `buildSystemGraph(systemId, nodes)` でシステム全体のグラフ構造を取得
- グラフはノード（AHU, DuctSegment, DuctFitting, Diffuser）とエッジ（ポート接続）で構成
- AHUをルートとした有向ツリーとして解釈

### 3. 🔵 葉ノード(Diffuser)のairflowRate取得 (REQ-802)

グラフの末端ノード（Diffuser）から設計風量を取得する。

- グラフの葉ノード（子を持たないノード）を特定
- 葉ノードがDiffuserであることを確認
- `diffuserNode.airflowRate` を取得（m³/h）
- 葉ノードがDiffuser以外の場合は警告ログ出力

### 4. 🔵 葉→根の逆トラバースで各区間風量を合算

深さ優先探索で葉から根に向かって風量を合算する。

- **アルゴリズム**:
  1. グラフの葉ノードを列挙
  2. 各葉ノードのairflowRateを取得
  3. 葉→親セグメントへ風量を伝播
  4. 分岐点（DuctFitting tee）では下流2枝の風量を合算
  5. 根（AHU）まで逆トラバース
- 各DuctSegmentの `airflowRate` に合算値を設定

### 5. 🔵 updateNode(各ductId, {airflowRate}) (REQ-801)

計算結果を各DuctSegmentNodeに反映する。

- 計算された風量値で `updateNode(ductId, { airflowRate })` を実行
- 変更があったノードのみ更新（不要な再描画を防止）
- 更新後、dirtyNodesから該当ノードをクリア

### 6. 🔵 AHU直近幹線のtotalAirflow検証 (REQ-803)

AHU直近のメインダクトの風量がAHU定格風量と整合するか検証する。

- AHU直近セグメントの airflowRate と AHU の airflowRate を比較
- 差異が許容範囲（例: ±5%）を超える場合に警告
- 警告はイベントバスで通知: `system:warning` イベント

### 7. 🔵 手動上書き時のdirty+再計算 (REQ-804)

ユーザーがDiffuserのairflowRateを手動変更した場合の再計算。

- DiffuserのairflowRate更新 → dirtyNodes に追加
- 次フレームで風量配分システムが再計算をトリガー
- 上流の全DuctSegmentの風量が再合算される

### 8. 🔵 サイクル検出時に中断+警告 (EDGE-004)

グラフにサイクルがある場合の安全処理。

- トラバース中に訪問済みノードを再訪した場合にサイクル検出
- サイクル検出時: トラバースを中断
- `system:error` イベントで「ダクト接続にループが検出されました」警告を発火
- 部分的に計算済みの風量はロールバックしない（次回修正時に再計算）

## 注意事項

- **コアシステムルール**: packages/core/src/systems/に配置。Three.jsインポート禁止、UI/レンダリング参照禁止
- **パフォーマンス**: 大規模システム（50ダクト以上）での再計算コストに注意。差分計算の検討は将来タスク
- **風量単位**: m³/h を基本単位とする。内部計算時にm³/sへの変換が必要な場合はユーティリティ関数で対応
- **手動上書きの優先**: REQ-804により手動設定値は自動計算より優先。ただしMVPでは手動上書きフラグの詳細管理は簡易実装
