右パネルにダクト/配管のプロパティを表示するコンポーネントと、配管のビジュアル更新システムを実装する。DuctPanelはダクト区間の断面形状・寸法・風量・圧損・材質・接続先を表示し、PipePanelは配管の口径・流量・流速・圧損・媒体種別を表示する。PipeVisualSystemは配管の色・太さをリアルタイム更新する。

## 実装詳細

### 1. 🔵 DuctPanel — ダクトプロパティパネル

**ファイル**: `packages/editor/src/components/sidebars/hvac/duct-panel.tsx`

ダクトセグメント選択時に右パネルに表示されるプロパティパネル。

- **表示項目**:
  - 断面形状: 矩形（rectangular）
  - 寸法: width × height (mm) — 編集可能
  - 風量: airflowRate (m³/h) — 表示のみ（自動計算結果）
  - 圧損: pressureLoss (Pa) — 表示のみ
  - 材質: material (galvanized / flexible) — ドロップダウン選択
  - 接続先ポート: startPortId / endPortId — 表示のみ
  - ダクト長: start→end距離 (m) — 表示のみ
- **入力操作**:
  - 寸法の手動入力 → updateNode(ductId, {width, height})
  - 材質変更 → updateNode(ductId, {material})

### 2. 🔵 PipePanel — 配管プロパティパネル

**ファイル**: `packages/editor/src/components/sidebars/hvac/pipe-panel.tsx`

配管セグメント選択時に右パネルに表示されるプロパティパネル。

- **表示項目**:
  - 口径: diameter (A表記、例: 50A) — 表示のみ（自動計算結果）
  - 流量: flowRate (m³/h) — 表示のみ
  - 流速: velocity (m/s) — 表示のみ
  - 圧損: pressureLoss (kPa) — 表示のみ
  - 媒体: medium (chilled_water_supply / chilled_water_return 等) — 表示のみ
  - 配管長: start→end距離 (m) — 表示のみ
- **入力操作**:
  - MVPでは基本的に表示のみ。口径・流量は自動算出

### 3. 🔵 PipeVisualSystem — 配管色・太さ更新

**ファイル**: `packages/viewer/src/systems/hvac/pipe-visual-system.tsx`

Viewerシステムとして実装。DuctVisualSystemと同様のパターンで、配管メッシュの色と太さをリアルタイム更新する。

- dirtyNodesにPipeSegmentが含まれる場合にビジュアル更新トリガー
- sceneRegistry.nodes から対応するObject3Dを取得
- **色更新**: medium変更時にカラー切り替え
  - chilled_water_supply: #0288D1
  - chilled_water_return: #01579B
- **太さ更新**: diameter変更時にメッシュスケール更新
  - radius = diameter / 2
  - 口径未設定（diameter=0）時は最小太さで表示
- **接続状態カラー**:
  - 両端接続済み: 媒体色（正常）
  - 片端未接続: 媒体色 + 透明度50%（接続不完全）
  - 口径未設定: グレー（#BDBDBD）

## 注意事項

- **Viewer Isolation**: PipeVisualSystemはpackages/viewerに配置。useEditorやphase/mode参照は禁止。DuctPanel/PipePanelはpackages/editorに配置
- **パネル共通化**: DuctPanelとPipePanelは類似構造。共通のPropertyFieldコンポーネント（ラベル+値表示/入力）を使いまわす設計を推奨
- **単位変換表示**: 内部値（m, m³/s）とUI表示値（mm, m³/h, A）の変換に注意。変換ユーティリティを用意
- **リアクティブ更新**: ノード更新時にパネル表示が即座に反映されるよう、useSceneのsubscribeを活用
- **PipeVisualSystemとDuctVisualSystemの統一**: カラー定数やdirtyNodes監視パターンは共通化を検討
